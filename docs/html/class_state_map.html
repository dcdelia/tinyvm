Detailed Description

Map program points and values from two functions.

This class is an essential component of the OSRKit library. A front-end can encode in a StateMap object correlation information between Value objects and between program points (i.e., Instruction objects) from two functions. OSRLibrary relies on StateMap to generate the entry block for OSR continuation functions.

typedef struct StateMap::CompCode StateMap::CompCode

Compensation code object.

This object can be used to encode local (i.e., required to reconstruct a specific Value) or global (i.e., to be executed at the beginning of the continuation function) compensation code.

A CompCode is essentially a lambda function that takes as arguments a subset of the live values transferred at an OSR point.

Express LocPair-specific mapping information for a Value.

Data structure used to express mapping information for a Value to set in the OSR target function. It is specific to a LocPair and contains either a CompCode object or a Value from the OSR source function that - although it does not provide a more general 1:1 mapping - can be still transferred to set this Value for the current LocPair.

Construct a StateMap object.

The constructed StateMap object can be either empty, or initialized using a ValueToValueMapTy (generated for instance by LLVM when cloning F2 from F1).

For each entry (k, v) in VMap, OSRKit assumes that k is a pointer to a Value in F1 and v is the corresponding Value in F2. This implies that when a continuation function is generated for an OSR from F1 to F2, each use of v in F2 can be replaced with the transferred value for k; we refer to this property as a 1:1 mapping, as the value to transfer from F1 to set v is the same for all the feasible pairs of OSR source locations and landing pads.

Typically, a 1:1 mapping is also bidirectional: that is, when generating a continuation function for an OSR transition from F2 to F1, each use of k can be replaced with the transferred value for v.

For each entry (k, v) in VMap, if both k and v are pointers to Instruction objects, OSRKit assumes that v is the landing pad for an OSR transition to F2 originated at k in F1. For bidirectional mappings, k also becomes the landing pad for an OSR from F2 to F1 originated at v.

When a VMap is not provided, the front-end can encode information on 1:1 mappings and landing pads using the registerOneToOneValue() and registerLandingPad() methods, respectively. In fact, OSRKit does not require a StateMap to be complete (i.e., containing information for all Value and feasible LocPair objects) in order to encode an OSR, but only to provide a landing pad in the target function for the OSR source location and to maintain information for live variables at the landing pad.

Finally, information for values for which there is no 1:1 corresponding value can be set using createLocPairInfo() and LocPairInfo::setValueInfo().

F2Second function for the mapping.

Generate entry block for an OSR continuation function.

The entrypoint will instantiate any global or local compensation code needed for the LocPair represented by OSRSrc and LPad. This method also generates a map - required by OSRLibrary::genContinuationFunc() - that associates values to set in the continuation function with either transferred values from the OSR source function, or with a value reconstructed in the entry point using local compensation code.

Parameters
ContextLLVM Context to use for OSRKit.
OSRSrcInstruction in the source function for which an OSR decision is taken.
LPadLanding pad in the target function for OSRSrc.
OSRContLPadInstruction in the OSR continuation function corresponding to LPad.
valuesToSetAtOSRContLive values to set in the continuation function to resume the execution.
fetchedValuesToOSRContArgsMap between values from the source function transferred at the OSR point and corresponding arguments of the OSR continuation function.
Returns
A pair where the first element is a BasicBlock suitable as entrypoint for the OSR continuation function, and the second element is a map between live values to set and arguments/reconstructed values.

Values in the sequence typically belong to the liveIn set at OSRSrc. In general, it is possible that a CompCode specific to the LocPair (OSRSrc, LPad) might also need Value objects not live at OSRSrc: in this case, OSRKit will artificially extend their liveness range.
